Privacy Policy

Last updated: April 2026

Who we are

Cookie is a self-hosted recipe management application. Your instance is operated by whoever deployed it. All data stays on the server where Cookie is installed — nothing is sent to us. Depending on how your instance is deployed, traffic may pass through infrastructure providers (such as Cloudflare for HTTPS, access control, and DDoS protection) or third-party APIs (such as OpenRouter for AI features, explained below). These providers process request data according to their own privacy policies.

What data we collect

In passkey mode

  • Username — automatically generated during registration (e.g. "User 1", "User 2"). You do not choose or provide a username — no identifying information is required to create an account. (source: core/passkey_api.py#L146)
  • Passkey credentials — a public key and credential ID generated by your device during passkey registration. These are cryptographic values, not biometric data. Your fingerprint or face scan never leaves your device. (source: core/models.py#L57 — WebAuthnCredential)
  • Profile preferences — display name, theme (light/dark), avatar colour. (source: profiles/models.py#L19)
  • Recipe data — recipes you import, favourites, collections, cooking history, and any AI-generated content (tips, remixes, serving adjustments). (source: recipes/models.py#L4)
  • Session cookie — a functional cookie to keep you logged in (expires after 12 hours of inactivity). Marked HttpOnly (prevents JavaScript access for XSS protection) and Secure (sent only over HTTPS). SameSite=Lax. (source: settings.py#L183)

In home mode (no authentication)

When Cookie runs without authentication (the default), only profile preferences and recipe data are stored. No usernames, passwords, or credentials of any kind.

What we do NOT collect

  • Email addresses — Cookie has no email field anywhere in its database. No authentication mode collects, processes, or stores email addresses. (verify: no EmailField in any model)
  • Passwords — passkey mode uses WebAuthn (biometric/hardware key authentication). There is no password entry, storage, or reset flow.
  • Tracking or analytics — no Google Analytics, Mixpanel, Segment, or any other tracking. No page view counts, click tracking, or user behaviour analytics.
  • Third-party scripts — the Content Security Policy header restricts scripts to self only. No external JavaScript is loaded. (source: nginx/security-headers.conf#L4)
  • Advertising data — no advertising identifiers, profiles, or remarketing pixels.
  • Location data — Cookie does not request or store your location.
  • Device fingerprints — Cookie does not fingerprint your browser or device. User-agent detection is used only to route legacy browsers (iOS 9, IE) to a compatible frontend. (source: core/middleware.py#L41)

Cookies

Cookie uses only two cookies, both strictly functional:

  • Session cookie (sessionid) — keeps you logged in. Expires after 12 hours. HttpOnly (not accessible to JavaScript), Secure (HTTPS only), SameSite=Lax.
  • CSRF cookie (csrftoken) — protects against cross-site request forgery attacks.

No tracking cookies, analytics cookies, or third-party cookies are set. Under UK PECR (Privacy and Electronic Communications Regulations), no cookie consent banner is required because all cookies are strictly necessary for the service to function.

Recipe search

When you search for recipes, your search query is sent to external recipe websites (such as AllRecipes, Food Network, and other configured sources) to fetch results. Only the search query text is sent — no username, session, or identifying information is included. Search results and their images are cached locally on the server to reduce repeat requests. (source: recipes/services/search.py#L54)

Images

Recipe images from external sources are downloaded and cached locally on the server when possible. In rare cases where local caching fails, images may be loaded directly from the original external source by your browser, which would create a request visible to that source. The Content Security Policy permits image loading from any HTTPS source to support this fallback. (source: recipes/services/image_cache.py#L40)

AI features and OpenRouter

If the site administrator configures an OpenRouter API key, Cookie can use AI for recipe remixes, serving adjustments, cooking tips, and discovery suggestions. When AI features are used:

  • Recipe text (ingredients, instructions, titles) is sent to OpenRouter's API for processing.
  • No personal data (username, preferences, browsing history) is included in AI requests.
  • AI responses are cached locally to reduce repeat API calls.
  • If no API key is configured, all AI features are completely hidden — not disabled, hidden. (source: AI features documentation)

Server logs

The web server (nginx) writes access logs that include request paths and user-agent strings. Depending on deployment, the logged IP address may be an internal proxy address rather than your real IP. These are standard web server logs used for debugging and are not shared with any third party. They are not used for tracking or analytics. Additional infrastructure components (reverse proxies, CDN providers, tunnel services) may also log request data according to their own policies.

Application logs never contain email addresses, passwords, or API keys. (source: core/logging.py#L8)

Data retention

  • Your account data is retained for as long as your account exists.
  • Device pairing codes expire after 10 minutes and are cleaned up automatically. (source: cleanup_device_codes.py#L12)
  • When you delete your account, all associated data (recipes, favourites, collections, history, AI-generated content) is permanently and irreversibly removed.
  • Administrators can delete individual user accounts and all their associated data.
  • Administrators can reset the entire database from Settings, which deletes all user data.

Legal basis for processing

Under UK GDPR (UK Data Protection Act 2018), data is processed on the basis of:

  • Consent — you consent to data processing by creating an account.
  • Legitimate interest — processing necessary to provide the recipe management service you have requested.

Your rights

Under UK GDPR, you have the right to:

  • Access — view all your data via the application interface (recipes, favourites, collections, history).
  • Rectification — update your theme and preferences from Settings.
  • Erasure — delete your account and all associated data from Settings.
  • Data portability — your recipe data is accessible through the application.
  • Object — contact the site administrator if you have concerns about data processing.

Data security

  • All connections are encrypted via HTTPS with HSTS (2-year max-age). (source: nginx/security-headers.conf#L1)
  • Authentication uses WebAuthn passkeys — no passwords to steal or brute-force.
  • API keys are encrypted in the database using Fernet symmetric encryption. (source: core/models.py#L14 — AppSettings)
  • Rate limiting protects against abuse on authentication and AI endpoints. Rate limits are enforced using your IP address, which is checked in memory but not stored or logged for this purpose. (source: settings.py#L204)
  • Content Security Policy restricts scripts, styles, and connections to same-origin only.
