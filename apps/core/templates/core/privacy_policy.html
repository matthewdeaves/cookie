Privacy Policy

Last updated: April 2026 (v1.72.2)

Who we are

This privacy policy covers the hosted Cookie instance operated by Matthew Deaves. The service runs on Amazon Web Services (AWS) in the London region (eu-west-2), with Cloudflare sitting in front for TLS, access control, and DDoS protection. The source code is open source; a separately-deployed copy you run yourself is outside the scope of this policy.

Cookie runs in passkey mode on this instance: accounts are anonymous and authenticated exclusively with WebAuthn passkeys.

What data we collect

  • Generated username — a random identifier of the form pk_xxxxxxxx (8 hex characters) created by the server during registration. You do not choose or provide a username, and no identifying information is required to create an account. (source: core/passkey_api.py#L164)
  • Display name — a randomly generated label (e.g. User a3f8c2) assigned when your profile is created. The label is not sequential and does not reveal how many accounts exist. The display name is fixed once assigned: end users cannot change it from the application UI. Only the operator can rename a profile, via the admin CLI; contact the operator (below) to request a change. (source: core/passkey_api.py#L182 — display name assigned; profiles/api.py#L305 — rename endpoint is HomeOnlyAuth (404 in passkey))
  • Passkey credentials — a public key and credential ID generated by your device during passkey registration. These are cryptographic values, not biometric data. Your fingerprint or face scan never leaves your device. (source: core/models.py#L65 — WebAuthnCredential)
  • Profile preferences — theme (light/dark) and avatar colour. (source: profiles/models.py#L19)
  • Recipe data — recipes you import, favourites, collections, recently-viewed recipe history, and any AI-generated content (tips, remixes, serving adjustments). (source: recipes/models.py#L4 — Recipe; recipes/models.py#L196 — RecipeViewHistory)
  • Session cookie — a functional cookie to keep you logged in (expires after 12 hours). Marked HttpOnly (no JavaScript access), Secure (HTTPS only), SameSite=Lax. (source: settings.py#L248 — SESSION_COOKIE_AGE = 43200 (12 hours))

What we do NOT collect

  • Email addresses — Cookie has no email field in its data model. The user record is created with an empty email string. (verify: user created with email="")
  • Passwords — authentication is WebAuthn-only. There is no password entry, storage, or reset flow.
  • Tracking or analytics — no Google Analytics, Mixpanel, Segment, or any other tracking. No page view, click, or behaviour analytics.
  • Third-party scripts — the Content Security Policy restricts scripts to self. No external JavaScript is loaded. (source: nginx/security-headers.conf#L4)
  • Advertising data — no advertising identifiers, profiles, or remarketing pixels.
  • Location data — Cookie does not request or store your location.
  • Device fingerprints — no browser or device fingerprinting. The User-Agent string is inspected only to route legacy browsers (e.g. iOS 9, IE) to a compatible frontend. (source: core/middleware.py#L84 — User-Agent inspected for legacy routing)

Cookies

Cookie uses only two cookies, both strictly functional:

  • Session cookie (sessionid) — keeps you logged in. Expires after 12 hours. HttpOnly, Secure, SameSite=Lax.
  • CSRF cookie (csrftoken) — protects against cross-site request forgery attacks.

No tracking cookies, analytics cookies, or third-party cookies are set. Under UK PECR (Privacy and Electronic Communications Regulations), no cookie consent banner is required because all cookies are strictly necessary for the service to function. Cloudflare may set its own operational cookies (for example __cf_bm for bot management) as part of its edge protection; see Cloudflare's privacy policy for details.

Recipe search

When you search for recipes, your search query is sent from the server to external recipe websites (such as AllRecipes, Food Network, and other configured sources) to fetch results. Only the query text is sent — no username, session, or identifying information is included. Search results and their images are cached on the server to reduce repeat requests. (source: recipes/services/search.py#L60 — server-side search method)

Images

Recipe images from external sources are downloaded and cached on the server when possible. In rare cases where local caching fails, images may be loaded directly from the original source by your browser, producing a request visible to that source. The Content Security Policy permits image loading from any HTTPS source to support this fallback. (source: recipes/services/image_cache.py#L33)

AI features and OpenRouter

AI features (recipe remixes, serving adjustments, cooking tips, discovery suggestions) are powered by OpenRouter. When you use them:

  • Recipe text (ingredients, instructions, titles) is sent from the server to OpenRouter's API for processing.
  • No personal data (username, preferences, browsing history) is included in AI requests.
  • AI responses are cached on the server to reduce repeat API calls.
  • OpenRouter routes the request to an underlying model provider; see OpenRouter's privacy policy for how they handle request data. (source: ai/services/openrouter.py#L34)

Server logs

Nginx writes access logs containing request paths and User-Agent strings. The client IP recorded is the one Cloudflare forwards via X-Forwarded-For. These logs are used for debugging and abuse investigation only, are stored on the AWS host, and are not shared with third parties. Cloudflare and AWS also generate their own edge and infrastructure logs under their respective privacy policies.

Application logs never contain passwords or API keys; the allowed structured fields are explicit. (source: core/logging.py#L25 — allowed structured log fields)

Data retention

  • Account data is retained for as long as your account exists.
  • Device pairing codes (used to log in legacy devices) expire after 10 minutes and are cleaned up automatically. (source: settings.py#L97 — 10-minute default; cleanup_device_codes.py#L28 — automatic deletion)
  • When you delete your account, all associated data (recipes, favourites, collections, history, AI-generated content) is permanently removed from the database.
  • Server and edge access logs are retained for a short operational window (weeks, not months) and then rotated out.

Legal basis for processing

Under UK GDPR (UK Data Protection Act 2018), data is processed on the basis of:

  • Consent — you consent to data processing by creating an account.
  • Legitimate interest — processing necessary to provide the recipe management service you have requested, and to protect the service from abuse.

Your rights

Under UK GDPR, you have the right to:

  • Access — view all your data via the application interface (recipes, favourites, collections, history).
  • Rectification — update your theme preference from Settings. Display name and avatar colour are fixed once assigned in passkey mode; contact the operator (below) to request a display-name change.
  • Erasure — delete your account and all associated data from Settings.
  • Data portability — your recipe data is accessible through the application.
  • Object — contact the operator (below) if you have concerns about data processing.

Data security

  • All connections are encrypted via HTTPS with HSTS (2-year max-age, includeSubDomains, preload). (source: nginx/security-headers.conf#L1)
  • Authentication uses WebAuthn passkeys — no passwords to steal or brute-force.
  • The OpenRouter API key is encrypted at rest in the database using Fernet symmetric encryption. (source: core/encryption.py#L29 — encrypt_value)
  • Rate limiting protects authentication and AI endpoints. The limiter uses your IP address (derived from the Cloudflare-forwarded X-Forwarded-For header) as an in-memory key; it is not stored or logged for rate-limiting purposes. (source: settings.py#L281)
  • Content Security Policy restricts scripts, styles, and connections to same-origin only.
